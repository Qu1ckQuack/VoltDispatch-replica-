Given the five very different roles in this system, generic UX advice (spacing scales, typography hierarchy) matters less here than getting the structural decisions right first. Here's what I'd prioritize, tied specifically to what we've already built:
1. Design five interfaces, not one interface with permission flags
HQ, Dealer, Coordinator, Technician, and Customer have almost nothing in common in how they'll use this. A single UI that shows/hides panels based on role tends to produce a mediocre experience for everyone — dense enough to overwhelm the Customer, sparse enough to frustrate HQ. Treat each role as its own information architecture, sharing only the underlying shadcn/ui component library and design tokens (same visual language, different structure) — this mirrors the module-boundary decision already made on the backend.
2. Match interface weight to how often the role uses it

No navigation to learn.
3. Make the WorkOrder state machine visible, not implicit
We already defined an explicit state machine (REQUESTED → ASSIGNED → ... → COMPLETED, with ISSUE/CANCELLED/RESCHEDULED branches). That should show up directly in the UI:

Customer/Dealer views: a visual stepper/progress tracker — gives an immediate mental model of "where is my order," instead of a vague status word
Technician/Coordinator views: only surface buttons for valid next transitions. Don't let someone tap "Complete" from "Assigned" — the UI should enforce the same state discipline the backend does, so invalid-state errors never even reach the API

4. Surface connection state explicitly — don't let real-time data go silently stale
Since location tracking and status updates depend on a WebSocket connection that can drop (technicians driving through dead zones), the UI needs a visible "Live" / "Reconnecting…" indicator with a last-updated timestamp. Without this, a dropped connection just looks like "the technician stopped moving" — which erodes trust in exactly the feature (live tracking) that's supposed to build it. This ties directly to the reconnect_hint mechanism from the backend deploy design — the frontend should visually reflect that same reconnect event.
5. Give every state transition a matching message, not just a status change
"Technician declined, reassigning" is a real event in your state machine — if the Dealer's UI just silently shows the status flip from ASSIGNED back to REQUESTED with no explanation, that reads as a bug, not a normal workflow step. Every non-happy-path transition (decline, cancel, issue, reschedule) needs an explicit, human-readable message, not just a badge color change.
6. Build trust for the Customer through specificity, not polish
A customer who's never used this system before trusts it more from concrete details — technician's name and photo, live ETA, a map with a moving marker — than from visual refinement. Prioritize showing real information over decorative design for that one screen; it's used once per order and needs to feel trustworthy in the first three seconds, not impressive.
